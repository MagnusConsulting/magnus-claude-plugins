Why this exists

The problem we set out to solve.

The Magnus site is built in Astro — fast, well-engineered, and componentised against a documented design system. That gives developers everything they need. It does not, on its own, give Magnus's non-technical administrators a way to make small content edits without breaking the system.

Without a tool, every routine update — a typo on a service page, a tweak to a CTA, a new article on /insights, a new team member — needs a developer. That has three consequences worth naming:

1 · Design system drift

An admin given direct access to the codebase can paste a hex value, an italic emphasis, or a one-off card style. Each instance is small. Over a year, the design system fragments and the brand starts to look inconsistent.

2 · Brand voice loss

Magnus copy is direct, evidence-led, short-sentence. Without rails, copy edits drift toward soft consultancy language — "synergies", "value-add", "solutions landscape" — that the firm has explicitly distanced itself from.

3 · Risk on legal pages and client metrics

The privacy policy, terms, and client growth-story metrics aren't routine admin edits. Changing a pipeline figure on a case study without partner sign-off is a real exposure. Without a system, there's no enforced check.

The toolkit replaces a CMS-shaped problem with a Claude-shaped solution: the admin describes what they want; Claude makes the change inside hard rails; the rails refuse anything that breaks the design system, the brand voice, or the approval rules.

Architecture

How it fits together.

Six pieces. Two repos, one contract document, a bundled MCP helper, an audit log, and a plugin marketplace that distributes the lot.

Layer 1 · Site

magnusconsulting-co-uk

The Astro website itself. Private GitHub repo. git push to the staging branch triggers CI, which auto-deploys to staging.magnusconsulting.co.uk. As far as the toolkit and admins are concerned, staging is the live site.

Layer 2 · Contract

docs/llm-context.md

A single document inside the website repo that defines the design system, component library, page templates, brand voice rules, and the team-member data shape. Every skill reads this document first as its source of truth — if it's missing, the skills refuse to act. This is the contract between the website and the toolkit.

Layer 3 · Plugin

magnus-claude-plugins

A separate private GitHub repo configured as a Claude marketplace. Contains the magnus-site plugin — a bundle of skills (admin-facing actions Claude can perform) plus a small local MCP helper. Versioned independently of the website. Distributed to admins via /plugin marketplace add.

Layer 4 · Helper

magnus-helper (bundled MCP)

A small Node script the plugin ships and auto-starts on enable. Runs on the admin's Mac (not in any sandbox) so Claude can start npm run dev, open URLs in the default browser, and bridge into Claude Cowork — where the standard Bash environment is sandbox-isolated.

Layer 5 · Audit

.magnus-changes/pending.log

A small log file inside the website repo, gitignored from production but read by the publish skill. Every gated change appends one line: timestamp, file path, summary, approver. The publish skill copies these into the commit message that lands on staging, then truncates the log.

Layer 6 · Distribution

Marketplace install

An admin runs two commands once: /plugin marketplace add ssu96ld/magnus-claude-plugins then /plugin install magnus-site. Updates propagate via /plugin marketplace update — or automatically on session start. No other setup.

Why two repos and not one? The website is a public-facing artefact with its own deploy pipeline, owned by the build team. The plugin is internal tooling that evolves on a different cadence. Keeping them separate means changes to the toolkit never accidentally land on the production website, and a new toolkit version can ship without touching the site.

The skills

What an admin can actually do.

Five skills today, each scoped to one well-defined admin action. Skills are triggered by natural-language requests — the admin doesn't memorise commands.

Skill What it does Files it touches Gate
magnus-setup
verify		 Detects whether the current Claude Project is pointed at the Magnus website folder. If yes, confirms readiness. If not, prints the canonical Terminal one-paste install (Homebrew + gh + Node + clone + npm install targeting ~/Projects/Claude/magnus-website) and tells the admin to re-open the project there. Never clones or installs from inside the Claude session — Claude's working directory is fixed at session start, so the install belongs in Terminal. none (read-only) none
magnus-edit-page
edit		 Edits copy, headings, CTAs, SEO metadata, or component props on an existing page. Refuses any change that introduces a new component, a raw colour, italic headings, or off-system spacing. Auto-navigates the live preview to the edited page. Pages under src/pages/ conditional
magnus-add-insight-article
create		 Creates a new article under /insights/<slug> using the canonical template, then inserts a matching ArticleCard at the top of the /insights grid with the right category, dataTags, and author colour. src/pages/insights/ always
magnus-add-team-member
create		 Appends a typed entry to src/data/team.ts. The dynamic route at /team/[slug] auto-generates the bio page; the team index surfaces the matching card. The skill never edits any .astro file. src/data/team.ts always
magnus-preview
preview		 Starts, navigates, or stops the local Astro dev server. Picks the right mechanism for the current Claude environment (preview pane, helper-driven browser, terminal Bash). Other skills call this hook automatically after applying changes. none (read-only) none

Each skill is defined as one Markdown file inside the plugin. The file specifies what the skill does, what it refuses to do, what approval it requires, and how it hands off to the next step in the workflow. Skills are easy to read and review — they aren't code so much as documented procedures.

Governance

Approval and audit are built in. Not bolted on.

The toolkit treats Magnus content as governed material. Some changes ship freely. Some require an explicit approval line in chat. Some never ship without partner sign-off. The boundaries are documented inside each skill — admins can't bypass them by phrasing things differently.

Routine edits — flow

A typo on /about, a CTA label change on a service page, a meta description tweak. Apply directly. No gate. Auditable in git history if anyone needs to look.

Path-gated edits — sign-off required

Legal pages (privacy-policy, terms-and-conditions, cookie-policy, modern-slavery-policy), the homepage hero, and any growth story under /impact/growth-story/. The skill stops, asks for approved by <name>, records the approval in the audit log.

Content-gated edits — always

Client metrics, real-money figures, percentages tied to a client name, client names in quotes, testimonial copy. Gated wherever they appear, on any page. Same approval mechanism, plus a second explicit line for the metric itself.

The audit log

Every gated change appends one line to .magnus-changes/pending.log in the website repo. The format is fixed:

2026-04-30T15:21:10Z | src/pages/impact/growth-story/dhl/strategic-accounts.astro | Update DHL pipeline metric from £270M to £290M | approved by Teresa Allan

The publish skill reads this log, copies every line into the commit message that lands on staging, and truncates the log file. The result is a clean audit trail: every gated change in the website's git history carries the name of the person who signed it off.

What the gate doesn't replace

The in-skill gate is a process check — it forces the admin to assert sign-off, and records the assertion. It doesn't replace partner judgement. The right way to think about it: the gate is the same friction as a "check before you commit" prompt, made auditable. Real review still happens with the partners themselves before the admin opens the chat — the gate just captures the assertion so it's traceable in the commit history.

Live preview

Three paths. Picked automatically.

Different Claude environments expose different capabilities. Cowork runs Claude in a remote Linux sandbox; solo Claude Desktop runs it on the user's machine; Claude Code in a terminal runs it as a local CLI. The preview skill picks whichever path the current session supports.

  1. Path A · Claude Desktop preview pane

    If the Claude_Preview MCP is wired (solo Desktop or Claude Code with the MCP installed), the dev server runs and renders inside the right-hand preview pane. Astro hot module reloading keeps it in sync with edits.

  2. Path B · magnus-helper bridge (Cowork)

    If the Claude_Preview MCP isn't available — typical of Cowork — the bundled magnus-helper MCP takes over. It runs on the admin's Mac (outside the sandbox), starts npm run dev, and opens the URL in the admin's default browser. Claude can navigate the browser by calling MCP tools through the user's account.

  3. Path C · System browser fallback (terminal)

    If neither MCP is available — Claude Code in a plain terminal session — the skill falls back to running npm run dev via Bash and opening the URL with open. Same outcome, different mechanism.

Why a cascade and not a single path?

Each environment has hard constraints. Cowork's sandbox can't reach the admin's local network, so the standard Bash fallback can't open localhost:4321 on the Mac. Solo Desktop has no need for the helper. Claude Code in a terminal doesn't have the Cowork bridge. The cascade means an admin gets a working preview in every environment without thinking about which one they're in.

Installing the toolkit on a new Mac

The admin runs three commands once. There is no other setup.

/plugin marketplace add ssu96ld/magnus-claude-plugins
/plugin install magnus-site
/reload-plugins

From that point, the admin can ask Claude — in plain English — to edit a page, add an article, add a team member, or preview the site. The skills handle the rest. Getting the website source onto the Mac is a one-paste Terminal install (Homebrew + gh + Node + gh repo clone + npm install targeting ~/Projects/Claude/magnus-website) done once during the quick-start; magnus-setup then verifies from inside Claude that the project is pointed at that folder.

Trust boundaries

What the toolkit will refuse. And what still needs a human.

The skills are deliberately narrow. Anything outside scope gets refused with a redirect to the right skill or to the developer.

Hard refusals

  • New components not documented in llm-context.md
  • Raw hex values or pixel sizes (must use design tokens)
  • Italic emphasis inside any <h1>–<h3>
  • Inline scripts, onclick handlers, third-party embeds
  • Italics or markdown inside team bios or article body
  • Edits to legal pages without an approval line
  • Client metrics or names in quotes without an approval line
  • Slug collisions on insights or team additions

Still requires a human

  • Photo and asset uploads — the toolkit doesn't yet handle binary files
  • Adding new components or design tokens — a developer change to the website repo and contract
  • The actual content sign-off from partners or the bio's subject before the admin asserts approval
  • Promoting staging further (e.g. to a separate production environment) — out of scope for the toolkit, owned by the build team
  • Adding new skills or capabilities to the plugin — a developer change to the marketplace repo

The trust model in one sentence

The toolkit makes routine edits safe and gated edits auditable. It does not make Magnus's content review automatic — partners still own what ships, the toolkit just makes it harder to ship something they shouldn't.

Roadmap

What's built. What's next. What's deferred.

The toolkit is shipped at v0.8.0 — the editing and content-creation loops are live, the preview cascade is working, and the audit log captures every gated change. One skill closes the publish loop; a second handles binary assets. After that, two more content-creation skills round out the inventory.

Built (v0.1.0 → v0.8.0)

  • magnus-setup — verify the Claude Project is pointed at the website repo, or print the Terminal install one-paste if not.
  • magnus-edit-page — edit existing pages with full guardrails.
  • magnus-add-insight-article — create new articles in /insights.
  • magnus-add-team-member — append entries to src/data/team.ts for the dynamic team profile pages.
  • magnus-preview — three-path preview cascade.
  • magnus-helper — bundled local MCP that bridges into Cowork.

Next (next iteration)

  • magnus-publish — closes the loop. Validates against the design contract, builds, commits to the staging branch with the audit log embedded in the message, and pushes. CI auto-deploys staging.magnusconsulting.co.uk on every successful push. Refuses to operate on any branch other than staging.
  • magnus-manage-assets — handles photo and document uploads to the website's public/ directory. Required for the team profile pages to show real photos rather than placeholders.

Deferred (when needed)

  • magnus-add-growth-story — large surface area, needs a careful pass over the GrowthStoryLayout data shape.
  • magnus-add-report — straightforward, similar to magnus-add-insight-article.
  • Ephemeral preview environments — per-change preview URLs for in-progress edits before they hit staging. Would eliminate the need for local npm run dev entirely; staging.magnusconsulting.co.uk already serves as the post-push preview.
Glossary

The terms that keep coming up.

Plain-English definitions for the bits of the architecture that aren't obvious from context.

Astro The framework the website is built in. Generates static HTML pages from .astro source files. Fast, well-suited to content sites, no runtime database.
Claude Cowork A feature of the Claude Desktop app where Claude runs in a remote Linux sandbox. Distinct from solo Desktop, where Claude has direct access to the user's machine. Cowork's sandbox is the reason the bundled helper MCP exists.
Claude Code Anthropic's command-line interface for Claude. Runs in a terminal. The toolkit works in Claude Code, Claude Desktop, and Cowork.
Plugin A bundle of capabilities Claude can install. The Magnus plugin is called magnus-site and contains the skills and the helper MCP.
Marketplace A catalogue of plugins. We run our own private marketplace (the magnus-claude-plugins repo on GitHub) so Magnus admins can install the toolkit by name.
Skill One documented admin action — "edit a page", "add an article", "add a team member". Each skill is a Markdown file describing what Claude does, what it refuses, and how it hands off.
MCP (Model Context Protocol) The protocol Claude uses to call out to external tools. Local MCPs run on the user's machine and are reachable from Cowork's sandbox via the user's account. The Magnus helper is a local stdio MCP shipped inside the plugin.
Slug The URL-safe identifier for a page or record — teresa-allan, gtm-confidence-index-2026. Always lowercase, hyphenated, no spaces.
Dynamic route An Astro page like src/pages/team/[slug].astro that generates one page per record in a data file. Used for team profiles — adding a row to team.ts creates a new page automatically.
Audit log A small text file (.magnus-changes/pending.log) inside the website repo where every gated change is recorded with timestamp, file, summary, approver. Read by the publish skill to compose the commit message that lands on staging.
Design contract The docs/llm-context.md file inside the website repo. Defines the design system, components, brand voice, and page templates. The skills read this on every invocation. Updating the contract is how you teach the toolkit about a new component or convention.
