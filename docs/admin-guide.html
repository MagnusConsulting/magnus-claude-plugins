User guide

Update the Magnus website. Just by talking to Claude.

A short guide to using the Magnus admin toolkit. No code. No CMS. You ask Claude — in plain English — to make a change. Claude does the work, shows you a preview, and ships it after you sign off.

Get set up

One-time install. Three commands.

Open Claude (Desktop, Cowork, or Code) and paste these commands into the chat — one at a time, pressing return after each.

/plugin marketplace add ssu96ld/magnus-claude-plugins
/plugin install magnus-site
/reload-plugins

That's it. Claude now knows how to edit the Magnus site. The first time you ask it to do anything, it'll check whether the website is already on your Mac — if it isn't, it'll offer to clone it for you and run the setup. Say yes.

Already done this?

Run /plugin marketplace update magnus-claude-plugins any time to pull the latest version of the toolkit. New skills and improvements arrive that way — no other action needed.

How it works

The loop is always the same. Four steps.

Whatever you're doing — fixing a typo, adding an article, publishing a new team member — the rhythm is the same.

Ask

Tell Claude what you want, in your own words. "Update the about page", "Add a new article", "Replace the homepage CTA". Claude figures out which skill to use.

Review

Claude shows you what it's about to change — the wording, the page, the file. You read it. If anything's off, you tell Claude and it adjusts. Nothing is saved yet.

Approve

Say "yes" or "go". For sensitive changes — legal pages, client metrics, team bios — Claude will ask you to paste an approval line so the sign-off is recorded.

Publish

Ask Claude to "ship it" or "publish". The change goes onto a branch, opens a PR for one final review, and goes live once that PR is merged.

Recipes

Common things you'll want to do.

Each recipe shows what to say, what Claude will do next, and what you'll need to have ready. Examples are starting points — phrase things naturally, Claude figures out the rest.

Edit something on an existing page

edit
Try saying

Fix a typo on /about — "consulutncy" should be "consultancy"

Update the headline on the home page to "Sparring partners. Not suppliers."

Change the primary CTA on /solutions/embed to point at /contact instead of /book

What Claude will do
  1. Confirm which page you mean and read it.
  2. Show you the exact change as a before/after.
  3. Refuse anything that breaks the design system — new components, raw colours, italic headings — and propose a compliant alternative.
  4. Apply the edit once you say yes, and open the page in your browser so you can check it.
You'll need
  • The page name or URL
  • The exact words to change
  • An approval line if it's a gated page (see below)

Publish a new article on /insights

create
Try saying

Add a new article on /insights — title is "Why GTM Is Now an Operating System", category GTM Strategy, by Teresa Allan, 7 min read

I have an article draft from Jason Ryan, can you publish it on /insights?

What Claude will do
  1. Ask for the title, lede, body, category, author, and SEO description if you haven't supplied them. You can paste a draft and Claude will pull out what it needs.
  2. Compose the full page using the canonical article template and add the matching card on the /insights index.
  3. Ask you to paste an approval line — articles always need a sign-off because they ship under the firm's voice.
  4. Save the page and open it in your browser.
You'll need
  • Title
  • Author (TA, DP, JR, or CD)
  • Category (GTM Strategy, AI in GTM, Private Equity, Sectors)
  • Body copy
  • SEO description (~150 chars)
  • Approval line

Add a new team member

create
Try saying

Add a new strategist — name is Frank Ocean, here's the approved bio…

I've got the bio doc for our new advisor. Can you add them to /team?

What Claude will do
  1. Read the bio you've supplied (paste it in chat or attach the document).
  2. Slot the person into the right group — Partners, Consultants & Strategy, AI Team, Operations & Capability, or Advisors & Associates.
  3. Show you the new entry, ask for the approval line, then save it.
  4. The bio page at /team/<slug> and the matching card on /team appear automatically.
You'll need
  • Approved bio (2–3 paragraphs)
  • Role / job title
  • Tagline (one sentence)
  • Related solutions and content (or "TBC")
  • Approval line

Publish a new client growth story

create
Try saying

Add a new growth story for DHL — strategic accounts, £290M pipeline in 16 weeks

Can you publish the Iron Mountain case study on /impact?

What Claude will do
  1. Ask you for the client name, headline metric, summary, the challenge / approach / outcome sections, and any pull quotes.
  2. Build the page using the standard growth-story template and add the matching card on /impact with the right tags so it shows up in the filter.
  3. Ask for two approval lines — one for the story itself, and one specifically for the client metric. Growth stories carry real client numbers, so they're double-gated.
You'll need
  • Client name and approval to publish
  • Headline metric (currency, percentage, etc.)
  • Story copy
  • Tags (sector, capability)
  • Two approval lines

What Claude will do
  1. Ask you to attach the file (drag-drop into chat, or paste a path).

Preview the site

preview
Try saying

Show me the site

Preview /team/teresa-allan

What Claude will do

Spin up the live development version of the site on your Mac and either show it inside Claude (Desktop) or open it in your default browser (Cowork). After that, anything you change will appear in the preview within a second or two.

Most of the time you don't need to ask for a preview yourself — the edit and create skills do it automatically after they save.

Publish your changes

ship
Try saying

Ship it

Publish the changes I just made

Open a PR for the team page updates

What Claude will do
  1. Run a quick check that nothing breaks the design system rules.
  2. Run the build to make sure the site still compiles.
  3. Create a branch, commit your changes with all the approval lines you gave inline in the message, and push.
  4. Open a pull request on GitHub for one final review, and hand you the URL.

The site goes live once that PR is merged. Claude never pushes straight to main — that's the safety net.

Approvals

When Claude asks for an approval line.

Some changes are routine — fix a typo, tweak a CTA. They go through. Others affect the firm publicly: legal copy, client numbers, team bios, growth stories. Those are gated. Claude will stop and ask you for an approval line before saving anything.

What's gated

  • The four legal pages (privacy, terms, cookie, modern slavery)
  • The homepage hero
  • Every page under /impact/growth-story/
  • Every new article on /insights
  • Every new team member or bio change
  • Anywhere that mentions a client name or a real-money figure
  • Testimonial and pull-quote copy

The exact phrase

Claude will tell you, but it's always this shape:

approved by <name>

For client metrics, you'll be asked for a second line that names the client and metric specifically:

approved by Teresa Allan for DHL pipeline metric

Paste the line in chat. Claude records it in the audit log and continues. The approver's name ends up in the commit message and the PR description, so you can prove sign-off later.

If you don't have approval yet — stop.

Don't paste an approval line you don't actually have. The toolkit assumes the approver has signed off. If they haven't, get the sign-off in the usual way (Slack, email, in person), then come back and paste the line. Five minutes of friction is the whole point.

Live preview

You'll see your changes in real time.

Whenever you make an edit or add something new, Claude opens the affected page so you can see the result immediately. What that looks like depends on which version of Claude you're using:

In Claude Desktop (solo)

The preview appears in the right-hand pane next to the chat. Edits hot-reload — you don't need to refresh.

In Claude Cowork

The preview opens in your normal default browser (Safari, Chrome, whatever you use). Keep the browser tab open and Claude will navigate it to the right page after each edit. If hot reload doesn't pick up a change, just hit refresh.

In Claude Code (terminal)

Same as Cowork — preview opens in your default browser. The chat stays in the terminal, the browser shows the site.

If you want to preview without making a change, just ask: "show me the site" or "preview /team". To shut it down: "stop the preview".

Publishing

What "ship it" actually does.

When you're happy with a set of changes, ask Claude to "ship it" or "publish". Here's what happens, step by step:

  1. Validation. Claude runs the design-system checker. If anything's off — a missing alt tag, a meta description out of bounds — it'll tell you and stop.
  2. Build. Claude runs the production build to make sure the site still compiles cleanly.
  3. Branch. A new branch is created off main (something like change/team-update-260501). Your changes are committed onto that branch, with every approval line you gave embedded in the commit message.
  4. Pull request. Claude opens a PR on GitHub and gives you the URL. The PR description lists every change and every approver — clear audit trail.
  5. Merge. A reviewer (developer or partner) opens the PR, reviews it, and merges it. The deploy fires automatically when the merge lands.

Why a PR and not a direct push?

It's the safety net. The toolkit tries hard to stop bad changes, but a second pair of eyes is cheap insurance against the cases nobody anticipated. The PR review is the last gate before the change is live.

Troubleshooting

If something goes sideways.

Most issues are one of these. Try the fix; if it still doesn't work, ask the build team.

"This folder isn't the magnus repo."

Claude is in the wrong directory. Say "run magnus-setup" and follow the prompts. It'll either find your existing copy or clone a fresh one.

"That change introduces a new component."

You're asking Claude to do something the design system doesn't cover. Either rephrase using a documented component (Claude will suggest the closest one) or send the request to the build team — adding new components is a developer change.

"That meta description is too short / too long."

SEO descriptions need to be 120–155 characters. Claude will offer a tightened version — accept it, or send it back with your own wording.

"The slug already exists."

You're trying to create something with the same URL slug as an existing page. Pick a different slug, or if you meant to update the existing page, tell Claude to "edit it instead".

"The dev server didn't start."

Usually means another tool is already using the port. Quit any other site previews and ask Claude to "restart the preview".

You changed your mind after Claude saved.

Tell Claude to "undo the last change" or "revert the team edit". As long as you haven't published yet, the change is local and easy to roll back. If you've already published, the build team can revert the PR.
