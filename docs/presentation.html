01
The big idea

No CMS. Claude is the web team.

There's no admin panel to log into. No drag-and-drop builder to learn. No tickets to file with developers for routine changes. You open Claude, describe what you want, and the site gets updated.

  • The website is code, not a database. It lives in GitHub as an Astro project. Pages, components, content — all version-controlled files.
  • Claude is your personal web designer. It knows the design system, the brand voice, the component library, and the publishing rules. It does the work.
  • You stay in plain English. "Fix that typo." "Add Frank's bio to the team page." "Publish the GTM Confidence Index." Claude figures out the rest.
  • The safety rails are baked in. Approval gates on legal pages and client metrics. Every change goes through a pull request. Nothing ships straight to main.

Talking point

The contrast is everything. WordPress, Webflow, Contentful — they all assume the bottleneck is a UI. The real bottleneck has always been institutional knowledge: brand voice, what goes where, who has to sign off. The toolkit encodes that knowledge so anyone can publish safely.

Step 2

Install the software you'll need.

Two pieces: the GitHub CLI (so the toolkit can read and write to the website repo on your Mac) and Claude itself. One Terminal command does the first job.

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)" \
  && brew install gh \
  && gh auth login

That installs Homebrew, then the GitHub CLI, then signs you in. ~5 minutes including download time.

Mac password once Press Return at the pause point "Login with a web browser" Authorize in your browser

Live demo

Open Terminal, paste the command, walk through the three pause points (sudo password, "press return to continue", GitHub auth flow). End by running gh auth status to show the green checkmark.

Claude itself

Any one works: Claude Desktop (Mac app), Claude Cowork (a mode inside Desktop), or Claude Code (terminal: npm install -g @anthropic-ai/claude-code). For the demo today we'll use [pick one].

03
Step 3

Set up Claude. Three commands.

The Magnus toolkit ships as a Claude plugin. It lives in a private marketplace on GitHub. To install: tell Claude where the marketplace is, install the plugin, reload.

/plugin marketplace add ssu96ld/magnus-claude-plugins
/plugin install magnus-site
/reload-plugins
  • Line 1 tells Claude where to find Magnus's plugins.
  • Line 2 installs the magnus-site plugin — that's the bundle of skills you'll use.
  • Line 3 reloads the session so the new skills become available.

Speaker note

This is the moment Claude transforms from a general assistant into a Magnus-specific tool. Once these three lines are done, every Claude session — anywhere — can edit the Magnus site. The plugin is yours per-account, not per-machine.

Updates later

To pull future improvements: /plugin marketplace update magnus-claude-plugins. New skills, fixes, voice rules — they arrive that way. No reinstall.

05
Step 5

Get the website onto your Mac.

All the skills assume Claude can see the website's source code. The first skill we run does exactly that — it clones the repo and installs the dependencies.

Say to Claude

Run magnus-setup

  • Claude checks if the website folder is already on your Mac. (It isn't, today.)
  • Offers to clone it. Say yes.
  • Default location is fine: a subfolder named magnusconsulting-co-uk.
  • Runs npm install behind the scenes — 30–60 seconds.
  • Asks you to start a fresh Claude session inside the cloned folder so the rest of the skills can find the code.

Why a fresh session?

Claude's working directory is fixed at session start. Once you're in the magnus folder, every other skill — preview, edit, add, publish — knows where it is and what it's working with.

Demo flow

Run setup → accept the clone → wait for npm install → close the chat → open a new Claude session in the new folder. Then we're ready to do real work.

Preview the site. Live.

Before we change anything — let's see the site running locally. The preview is hot-reloading: every edit you make from here on shows up in seconds, no refresh.

Say to Claude

Show me the site

Preview the team page

07
Step 7

Edit something on a page.

The most common task. Fix a typo, swap a headline, change a CTA destination, tweak an SEO description. The edit skill handles all of it.

Say to Claude

Fix the typo on /about — "consulutncy" should be "consultancy"

Update the headline on the home page to "Sparring partners. Not suppliers."

Change the primary CTA on /solutions/embed to point at /contact

  • Claude finds the page, reads it, and shows you a before/after diff in chat.
  • If the change would break the design system — a new component, a raw colour, an italic heading — Claude refuses and offers a compliant alternative.
  • Once you approve, the file is saved. The preview navigates to that page automatically.
  • Nothing is live yet. The change is local on your Mac until you publish.

Live demo idea

Pick something visible — the about page hero, a CTA, the meta description on a page someone in the room has been complaining about. Watch the preview update in real time as Claude saves the change.

08
Step 8

Add a new insight article.

A one-shot skill. From a draft in an email or a quick verbal brief, Claude composes the full page, drops it into /insights/<slug>, and adds the matching card on the index.

Say to Claude

Add a new article on /insights — title is "Why GTM Is Now an Operating System", category GTM Strategy, by Teresa Allan, 7 min read

I have an article draft from Jason — can you publish it on /insights?

Claude will collect what it needs:

Title Author (TA, DP, JR, CD) Category Lede Body SEO description
  • The page is generated from the canonical article template — typography, layout, related-content all on-brand.
  • An ArticleCard is inserted at the top of the /insights grid automatically.
  • Categories are constrained to the four documented ones: GTM Strategy, AI in GTM, Private Equity, Sectors.
  • Authors are constrained to the four known initials. New author = developer change.

Demo prompt

Have a real (or made-up) article draft ready before the talk. Paste it in chat and let Claude pull title, author, category, body, SEO out of it. The audience watches the article appear in the preview.

09
Step 9

Approval gates. Why some changes pause.

Most changes go straight through. Some don't — they stop and wait for an explicit sign-off line in chat. That's intentional. The toolkit refuses to ship sensitive copy without a named approver attached.

What's gated

  • The four legal pages (privacy, terms, cookie, modern slavery)
  • The homepage hero block
  • Every page under /impact/growth-story/
  • Every new article on /insights
  • Every new team member or bio change
  • Anywhere a client name, real-money figure, or testimonial appears
The exact phrase

approved by Teresa Allan

approved by Teresa Allan for DHL pipeline metric

  • You paste the line in chat. Claude records it in .magnus-changes/pending.log.
  • The approval travels with the change — into the commit message and the PR description.
  • Audit trail: months later, anyone can see who signed off on what.
  • Client metrics are double-gated — one approval for the story, one for the number.

The honest framing

Don't paste an approval you don't actually have. Five minutes of friction is the whole point — it stops a 9pm "quick edit" turning into a public correction the next morning.

10
Step 10

Publish to live.

When you're happy with the preview, ship it. Two words. Claude takes care of every step from your laptop to a pull request waiting on GitHub.

Say to Claude

Ship it

Publish the changes I just made

  1. Validate. Run the design-system checker. Anything off — a missing alt tag, a meta length out of bounds — and Claude stops to fix it first.
  2. Build. Run the production build to confirm the site still compiles.
  3. Branch & commit. A new branch off main with every approval line embedded in the commit message.
  4. Push & PR. Claude opens a pull request on GitHub and hands you the URL. The PR description lists every change and every approver.
  5. Merge. A reviewer checks it, merges, and the site auto-deploys.

Why a PR — not a direct push

It's the safety net. The toolkit blocks most bad changes; a second pair of eyes catches the rest. Branch protection on main means it's not optional — every change goes through review before it goes live. Five-minute round trip; zero "how did that get on the site" moments.

Closing demo

Run "ship it" on the changes you've made during the session. Show the PR URL Claude gives you, click into GitHub, walk through the commit message and PR description with the approval lines embedded. That's the full loop.

That's the system
